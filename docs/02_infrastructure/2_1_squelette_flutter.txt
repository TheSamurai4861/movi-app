Point 2.1 – Créer le squelette Flutter
======================================

Objectif général
----------------
Mettre en place une architecture Flutter robuste qui sépare clairement les responsabilités entre les modules `core` et `features`, facilite la navigation et prépare les fondations de theming et de gestion d’état. Cette étape vise à poser les bases techniques nécessaires pour itérer rapidement sur les écrans ultérieurs tout en maintenant une qualité de code élevée.

Livrables clés
--------------
1. Arborescence standardisée du dossier `lib/` avec séparation `core/` et `features/`.
2. Modules transverses (routing, thème, dépendances, utilitaires) initialisés dans `core/`.
3. Modules `features/` précréés avec placeholders pour les pages principales (home, search, library, settings, movie, tv, saga, playlist, person).
4. Configuration initiale de la navigation (GoRouter) et injection de routes basiques.
5. Mise en place du thème global (light/dark, typographies, colorScheme) et scaffolding des widgets partagés.
6. Documentation minimale décrivant la structure et les conventions de nommage.

Étapes détaillées
-----------------

1. Analyse et préparation
   1.1 Lister les pages principales, les composants transverses et les modules utilitaires prévus par la roadmap. OK
   1.2 Identifier les dépendances nécessaires (go_router, get_it ou autre DI, provider/bloc, packages UI). OK
   1.3 Définir la nomenclature des fichiers/dossiers (snake_case pour les fichiers, camelCase pour les classes, usage de suffixes `_page`, `_view`, `_controller`, `_repository`, etc.). OK
   1.4 Mettre à jour la documentation projet (README ou docs internes) pour référencer les conventions retenues. OK

2. Structuration des dossiers
   2.1 Créer l’arborescence suivante dans `lib/src/` :
       - `app.dart` et `main.dart` pour l’entrée d’application.
       - `core/` contenant `router/`, `theme/`, `di/`, `utils/`, `widgets/`. OK
       - `features/` avec un sous-dossier par domaine (`home/`, `search/`, `library/`, `settings/`, `movie/`, `tv/`, `saga/`, `playlist/`, `person/`). OK
       - À l’intérieur de chaque feature : `presentation/` (pages, widgets), `data/` (repositories, sources), `domain/` (entités, usecases) même si certains restent vides pour l’instant. OK
   2.2 Ajouter des fichiers README.md locaux pour décrire la responsabilité de chaque dossier si nécessaire. OK
   2.3 Vérifier la cohérence avec le tooling (VSCode, IntelliJ) et adapter les configurations `.vscode/` ou `.idea/` si besoin. OK

3. Initialisation des modules core
   3.1 `core/router/` : créer `app_router.dart` avec `GoRouter`, centraliser les routes et définir une classe `AppRouteNames`. OK 
   3.2 `core/theme/` : créer `app_theme.dart` avec `ThemeData` light/dark, palettes de couleurs, textes, formes, `ThemeExtension` éventuelles. OK
   3.3 `core/di/` : mettre en place un injecteur (ex. `GetIt` ou `Riverpod ProviderContainer`) pour enregistrer les services partagés (API client, cache, repositories). OK
   3.4 `core/utils/` : ajouter des helpers de base (constantes, extensions) à utiliser partout dans l’app. OK
   3.5 `core/widgets/` : poser les premières briques réutilisables (boutons, cartes, loaders, placeholders).
   3.6 Documenter le point d’entrée (`app.dart`) qui assemble thème, routeur et localisation.

4. Mise en place des features
   4.1 Pour chaque feature, créer :
       - `presentation/pages/<feature>_page.dart` avec un widget minimal (Scaffold + placeholders).
       - Un sous-dossier `widgets/` pour lister les composants spécifiques.
       - Des dossiers `data/` et `domain/` avec un fichier README ou un fichier vide prêt à accueillir les implémentations futures.
   4.2 Ajouter des tests de squelette (ex. tests de rendu de widget minimal) pour vérifier que chaque page se construit sans erreur.
   4.3 Prévoir des hooks pour l’état (ex. `ChangeNotifier`, `Cubit`, `Riverpod` provider) même si l’implémentation reste vide.

5. Configuration de la navigation
   5.1 Enregistrer toutes les routes principales dans `app_router.dart` via `GoRoute`.
   5.2 Définir des noms de routes constants pour faciliter la navigation et l’internationalisation future.
   5.3 Implémenter un `errorPageBuilder` ou une route fallback pour gérer les chemins inconnus.
   5.4 Configurer les redirections initiales (ex. route par défaut, onboarding si nécessaire).
   5.5 Ajouter des tests de navigation (ex. `GoRouter` testing) pour valider la mise en place.

6. Thème et configuration globale
   6.1 Intégrer `AppTheme` dans `MaterialApp.router` (ou `Builder` principal) pour appliquer le thème.
   6.2 Définir les couleurs principales en accord avec l’identité visuelle de MOVI.
   6.3 Élaborer les styles de texte (headline, title, body, label) pour assurer la cohérence UI.
   6.4 Créer des composants communs (ex. `MoviScaffold`, `MoviSection`) si nécessaire pour homogénéiser les écrans.
   6.5 Prévoir les themings spécifiques (dark mode, contrastes) et l’internationalisation (localizations delegates).

7. Documentation et communication
   7.1 Décrire la structure dans `docs/movi_project_roadmap.txt` ou un fichier dédié.
   7.2 Mettre à jour les diagrammes d’architecture si disponibles (ex. module diagram, routing map).
   7.3 Partager les conventions de code dans le wiki ou la doc interne (lint rules, patterns à suivre).
   7.4 Organiser une revue d’architecture avec l’équipe pour valider ou ajuster les choix.

8. Validation et qualité
   8.1 Exécuter `flutter analyze` et `dart format` pour garantir la conformité.
   8.2 Lancer les tests de squelette pour s’assurer que chaque page se construit sans exceptions.
   8.3 Configurer l’intégration continue minimale (ex. GitHub Actions) pour lancer analyse et tests à chaque PR.
   8.4 Vérifier que les dépendances sont verrouillées (`pubspec.lock`) et qu’aucun fichier généré n’est commité par erreur.
   8.5 Préparer une checklist d’onboarding pour les développeurs rejoignant le projet (structure, commandes, scripts).

9. Prochaines étapes après 2.1
   9.1 Remplir progressivement les features avec la logique métier (points 3 à 10 de la roadmap).
   9.2 Introduire la vraie gestion d’état (bloc, Riverpod, etc.) et les repositories concrets.
   9.3 Connecter les données mockées ou API réelles pour valider la structure.
   9.4 Renforcer la documentation (diagrammes de navigation, guides contributeurs).
   9.5 Planifier les améliorations transverses (tests automatisés, performance, accessibilité).
