## Règles

- N’ajoute jamais de clé technique, temporaire, issue d’un outil ou d’un widget.
- Réutilise une clé existante si elle couvre déjà le besoin métier.
- Sinon, crée une clé avec un nom explicite en camelCase, préfixée par la feature.
- Ajoute la clé dans app_en.arb puis dans toutes les locales.
- Garde exactement les mêmes placeholders et métadonnées utiles dans toutes les locales.
- N’introduis jamais de hc_*, de noms vagues, de duplicats conceptuels ou de texte mixant technique et produit.
- Écris chaque valeur comme un vrai texte d’interface, pas comme une traduction brute ou un extrait de code.
- Si le contexte est ambigu, ajoute une description dans @key.
- Avant validation, vérifie parité, JSON valide, placeholders, doublons, cohérence de ton.

## Choix

- Choix de langues largement suffisant voire trop grand.
- Je dois vérifier dans chaque fichier si des localizations sont manquantes
