# Problématiques par fichier

## 82 — `src/core/widgets/overlay_splash.dart`

* **Bug de syntaxe** dans la condition qui affiche le message : ton export montre `if (message != null && message!.isNotEmpty) .[ … ]` au lieu du spread `...[` → ça casse la compilation / hot-reload de manière intermittente (selon l’éditeur) et peut expliquer un overlay « figé » s’il n’est pas reconstruit correctement. 
* **Pas de fade interne** : l’overlay est « dur » ; tout le fade est géré par la page. Ce n’est pas bloquant, mais un `AnimatedOpacity` interne te donnerait un “fail safe” visuel s’il est ré-utilisé ailleurs. 

## 86 — `src/features/home/presentation/pages/home_page.dart`

* **Retrait différé de l’overlay** : tu fais bien le fade + retrait du tree via `_overlayMounted`, mais l’état de l’overlay n’est **jamais ré-initialisé** sur un retour depuis d’autres onglets ou un pull-to-refresh complet (si tu veux un overlay « de session », c’est OK ; sinon il faut prévoir la ré-apparition sur relance). 
* **Empilement / z-order** : l’overlay couvre la bottom-bar (volontaire), mais sur appareils lents un `onEnd` de l’`AnimatedOpacity` peut sauter si le widget est disposé trop tôt. Tu mitiges avec `_overlayMounted`, mais garde en tête qu’un rebuild parent agressif peut court-circuiter le callback. 

## 89 — `src/features/home/presentation/widgets/home_hero_section.dart`

* **Plafond de décodage non appliqué** : tu déclares `_maxHeroCachePx = 1440` mais tu **n’appliques pas** de `cacheWidth/cacheHeight`/`ResizeImage` à l’image du backdrop. Sur Windows, un resize de fenêtre peut provoquer un **décodage géant** → frame bloquée et overlay qui tarde (exactement ton symptôme). 
* **Signal “ready” du backdrop fragile** : selon l’implémentation (tronquée dans l’extrait), si tu relies `onBackgroundReady` à un simple build/resolve sans `frameBuilder/ImageStreamListener`, tu peux notifier trop tôt (avant frame peinte) ou jamais (en cas d’erreur). Ton résumé indique un **fallback ready même en erreur**, mais je ne le vois pas encore dans le code → à sécuriser côté Hero. 
* **Hydratation** : la logique cache → full network est saine, mais si `vote/overview/logos` manquent, la ré-hydratation peut retarder la première frame si tu re-évalues juste au moment du build (setState immédiat). Un **déclenchement différé** (post-frame) évite de bloquer la peinture initiale. 

## 76 — `src/core/widgets/movi_items_list.dart`

* **Calcul de viewport** : tu sauvegardes une largeur `_lastViewportWidth` recalculée depuis `LayoutBuilder` puis **re-soustrais** le padding plus tard (OK), mais c’est **sensible au mismatch** si le padding change dynamiquement (RTL, tailles adaptatives). Pas bloquant, mais source de notifications de viewport « flottantes ». 
* **Hook vertical** : l’écoute de `isScrollingNotifier` du parent est bien nettoyée, mais si la liste est rendue dans un `CustomScrollView` ré-attaché, tu peux **manquer** une première notif (d’où l’ajout que tu as fait : enrichissement initial post-layout — très bien). 

## 87 — `src/features/home/presentation/providers/home_providers.dart`

* **Back-pressure** : `_maxConcurrent = 6` + flush 64 ms Windows, top. Mais `cancelOffscreen` annule uniquement **hors fenêtre** : en cas de défilement très rapide, il peut rester des requêtes « ancien viewport mais partiellement chevauchées ». Tu le limites via `_lastReq.contains`, mais une **clé “generation”** par section réduirait encore les races. 
* **load()** : Hero d’abord, listes ensuite avec `48ms` de respiration — très bien. Prévoir un **reset propre** de l’overlay côté page si tu souhaites rejouer l’expérience au prochain `load()`. 

## 84 — `src/features/home/data/repositories/home_feed_repository_impl.dart`

* **Types dynamiques** : `_fetchTrendingMoviesPage` renvoie `List<dynamic>` puis `_mapMovie` suppose des DTO typés (`dto.posterPath`, etc.). Si un fallback renvoie un `Map`, on obtient un **runtime error**. Il faut **resserrer les types** ou un garde robuste. 
* **Search fallback** (enrichissement) : bien vu (`title → id`). Sur Web tu forces l’enrichissement si poster http → ok. Rien de bloquant ici.

1. **Appliquer `_maxHeroCachePx`** : construire le backdrop via `ResizeImage` / `cacheWidth` pour plafonner le décodage à ~1440 px (côté plus long). Gain net CPU/RAM sur Windows lors des resizes.

3. **Hydratation non bloquante** : si des champs essentiels manquent (`logos`, `overview`, `vote`, `runtime`), déclenche le fetch full mais **décale le setState** à l’issue (ou en post-frame) pour ne pas retarder la première peinture.

3. Logguer l’issue « soft-timeout hit » pour confirmation terrain.
