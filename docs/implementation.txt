Plan d’implémentation — Feature dédiée pour la page Playlist

Objectif
- Extraire la logique d’affichage et de récupération des éléments d’une playlist (catégorie) hors de la feature `playlist`, en créant une feature dédiée, réutilisable et conforme à la Clean Architecture.
- La page `PlaylistDetailPage` devient un hôte (shell) qui consomme la nouvelle feature de navigation/affichage des contenus d’une catégorie.

Nom de la feature
- `category_browser` (navigation et affichage des contenus d’une catégorie/playlist)

Architecture (Clean)
- domain
  - Entities: `Category`, `CategoryItem` (id, title, poster, type, year?, rating?)
  - Value objects: `CategoryKey`, `Title`
  - Repository contracts: `CategoryRepository` (obtenir liste par `CategoryKey`, pagination, recherche facultative)
  - Use cases: `GetCategoryItems`, `PaginateCategoryItems`, `ObserveCategoryItems` (si streaming)
  - Erreurs typées: `CategoryFailure` (network, parse, empty, unknown)
- data
  - Data sources: `CategoryRemoteDataSource` (client HTTP/IPTV), `CategoryCacheDataSource` (cache mémoire optionnel)
  - DTOs + Mapper: `CategoryItemDto` → `CategoryItem`, parse centralisé
  - Implémentations: `CategoryRepositoryImpl` (map exceptions → failures, timeouts/retry configurables)
  - Config: endpoints, timeouts, headers via DI
- presentation
  - State: `CategoryState` (loading, data, error, page, hasMore)
  - Controller: `CategoryController` (Riverpod Notifier), orchestre use cases
  - Widgets: `CategoryGrid`, `CategoryHeader`, `CategoryErrorView`
  - Provider: `categoryControllerProvider(CategoryKey)` (family)

Étapes d’implémentation
1) Domain
   - Créer `CategoryItem` et `CategoryKey` (immutables, null-safety stricte).
   - Définir `CategoryRepository` (contrats: `Future<Result<List<CategoryItem>, CategoryFailure>> getItems(CategoryKey, {page, pageSize})`).
   - Ajouter use cases `GetCategoryItems`, `PaginateCategoryItems` (appellent le repo, renvoient `Result`).

2) Data
   - Ajouter `CategoryItemDto` et `CategoryItemMapper`.
   - Implémenter `CategoryRemoteDataSource` (HTTP client injecté via DI, timeouts/retry paramétrables, pas de retry infini).
   - Implémenter `CategoryRepositoryImpl` (map DTO → domain, map exceptions → `CategoryFailure`).
   - Optionnel: `CategoryCacheDataSource` (mémoire, TTL court) pour UX.

3) Presentation
   - Créer `CategoryState` (data, loading, error, page, hasMore).
   - Créer `CategoryController` (Riverpod Notifier) utilisant les use cases; expose `fetchFirstPage()`, `fetchNextPage()`.
   - Créer `categoryControllerProvider = NotifierProvider.family` (clé de catégorie en paramètre).
   - Créer `CategoryGrid` (2 colonnes, centrée, lazy) et `CategoryHeader` (titre + bouton retour aux positions spécifiées).

4) Intégration dans PlaylistDetailPage
   - Remplacer l’accès aux providers de home par `categoryControllerProvider(args.categoryKey)`.
   - Injecter les dépendances via DI (client HTTP, repo) au niveau de l’app.
   - Conserver le layout (bouton retour à 55px/20px, grille centrée 2 colonnes, gap 24px vertical).

5) DI & config
   - Enregistrer `CategoryRepositoryImpl` et `CategoryRemoteDataSource` dans le container (Riverpod providers).
   - Centraliser base URL/IPTV endpoints et timeouts dans un `CoreConfig`.

6) Tests (sur demande uniquement)
   - Mapper: DTO → Model
   - Repository: succès/échecs (network, parse)
   - Controller: pagination et états

Règles appliquées (qualité)
- Clean Architecture stricte: contrats `Repository` dans domain, implémentations dans data.
- Parsing uniquement en data, types stables en sortie.
- Gestion d’erreurs typée (Result/Failure), pas d’exceptions qui remontent en UI.
- DI obligatoire (client HTTP, config, repo), pas d’instances globales cachées.
- Null-safety, immutabilité par défaut, lints respectés.
- Logs sobres, pas de payloads sensibles.

Livrables attendus
