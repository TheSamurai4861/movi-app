Plan d’implémentation — Refonte Bienvenue & Page Recherche (aligné Clean Architecture)

Préambule (règles projet)
- Couches séparées (domain/data/presentation) et dépendances uniquement vers le bas.
- Contrats `Repository` en domain, implémentations en data. Parsing/DataSource dans data.
- Erreurs typées (Failure/Result), DI via injector, null-safety stricte.
- UI sobre, pas de préchargement massif, pagination/streaming si nécessaire.

1) Refonte page Bienvenue (Inscription en 2 étapes)
- Objectif: recueillir profil utilisateur (prénom, langue, métadonnées) puis guider vers l’ajout de source IPTV.

1.1 Domain
- Entity: `UserProfile { firstName, languageCode, metadataPrefs }`.
- Value objects: `FirstName`, `LanguageCode`, `MetadataPreference` (validation/coercion).
- Repository contract: `UserSettingsRepository { Future<void> save(UserProfile), Future<UserProfile?> load() }`.
- Use cases: `SaveUserProfile`, `LoadUserProfile` (retours typés `Result<void, Failure>` et `Result<UserProfile?, Failure>`).

1.2 Data
- DataSource local: `UserSettingsLocalDataSource` (stockage clé-valeur; sérialisation VO). Pas de JSON en domain.
- Impl: `UserSettingsRepositoryImpl(dataSource)` avec mapping exceptions → `Failure` (storage).
- DI: enregistrer `UserSettingsLocalDataSource` et `UserSettingsRepositoryImpl` dans `injector.dart`.

1.3 Presentation — Étape 1: Page user
- UI: titre “Bienvenue !” + description courte (mise en page similaire “ajout source”).
- Champs: input prénom, dropdown langue app, dropdown métadonnées.
- Règles UI: espacement cohérent (ex: 20px latéraux), validations VO inline.
- Actions: bouton “Continuer” → appelle `SaveUserProfile` puis navigation Étape 2.

1.4 Presentation — Étape 2: Page ajout source
- Réutiliser la page existante d’ajout de source (IPTV connect); route `iptv_connect`.
- Pré-remplir si des infos contextuelles existent; sinon flux classique.

1.5 États/Providers
- `UserSettingsController` (StateNotifier) avec `UserSettingsState { profile, isSaving, error }`.
- Providers Riverpod: `userSettingsRepositoryProvider`, `userSettingsControllerProvider`.

1.6 Qualité & sécurité
- Logs sobres (info au save, error sur storage). Pas de payload sensible en clair.
- Defaults sûrs: si `load()` renvoie null, afficher champs vides mais sûrs.

1.7 Étapes de livraison
- Créer contrats domain + VO + use cases.
- Implémenter data source local + repository impl + DI.
- Construire pages UI Étape 1 et 2 + providers + navigation.
- Test manuel: cycle save/load, reprise app (profil rechargé), puis connexion source.

2) Page Recherche (films & séries + historique lié user)
- Objectif: rechercher films/séries, auto-search après 1s d’inactivité, min 3 lettres, prise en charge `tmdb` direct via `txxxxxxx`, historique persistant.

2.1 Domain
- Value object: `SearchQuery` (min 3 lettres, detection `tmdbPattern = /^t\d{7,}$/` → mode ID direct).
- Entities existantes: `MovieSummary`, `TvShowSummary` (réutiliser types stables de l’app).
- Repositories contracts:
  - `SearchRepository { searchMovies(SearchQuery), searchShows(SearchQuery) }`.
  - `SearchHistoryRepository { add(query), list(), remove(item) }`.
- Use cases: `SearchMovies`, `SearchShows`, `AddSearchQueryToHistory`, `ListSearchHistory`, `RemoveSearchHistoryItem`.

2.2 Data
- Impl existante: `SearchRepositoryImpl` (ajuster pour reconnaître `SearchQuery.mode` = text vs `tmdbId`).
- Filtrage disponible: conserver filtre `availableTmdbIds` (via `IptvLocalRepository`).
- DataSource histoire: `SearchHistoryLocalDataSource` (liste ordonnée par récent, persistance clé-valeur). Impl `SearchHistoryRepositoryImpl` (mapping exceptions → Failure).
- DI: enregistrer `SearchHistoryLocalDataSource` + `SearchHistoryRepositoryImpl`.

2.3 Presentation — UI layout
- Header: haut gauche, titre 24px semibold “Recherche” (padding top 0, left 20).
- 16px en dessous: input texte placeholder “Tapez votre recherche”.
  - Trailing: `assets/icons/search.png` 25×25.
  - Si texte présent: bouton suppression `assets/icons/suppression.png` 25×25 en fin.
- 32px en dessous:
  - Si aucune recherche encore: section “Historique” (titre medium 20px) + liste verticale largeur `screen - 40px`.
    - Item: row 55px, align center, space-between: titre (regular 16px) + bouton suppression (icon 25×25). Supprime et recharge liste.
  - Sinon si résultats: deux sections MoviItemsList
    - “Films” + sous-titre “(X résultats)” → résultats triés par pertinence.
    - “Séries” + sous-titre “(Y résultats)”.
  - Sinon: texte centré 20px medium “Pas de résultats”.

2.4 Comportement & contrôleurs
- Déclenchement: auto recherche après 1s sans saisie (debounce). Min 3 lettres.
- Mode `tmdb`: si `SearchQuery.isTmdbId`, bypass texte, requête directe par ID.
- Historique: ajouter la requête au succès, afficher liste au repos. Suppression rafraîchit provider.
- Providers Riverpod:
  - `searchControllerProvider` (state: { query, movies, shows, isLoading, error }).
  - `searchHistoryControllerProvider` (state: { items, isLoading }).

2.5 Qualité & perfs
- Débounce 1s, pas de spam réseau; pagination optionnelle si API la supporte.
- Logs: `debug/info/error` sans données sensibles.
- Pas de préchargement massif d’images; réutiliser `MoviItemsList`.

2.6 Étapes de livraison
- Domain: créer `SearchQuery` + contrats `SearchHistoryRepository` + use cases.
- Data: impl data source local + repository impl + DI.
- UI: construire `SearchPage` selon layout; brancher providers; intégrer suppression de l’historique.
- Validation manuelle: auto-search, min 3 lettres, mode tmdb, historique (add/list/remove), affichages Films/Séries.

2.7 Voir tout (films/séries) + limite 10 sur Recherche
- Affichage Recherche: limiter l’affichage à 10 éléments par section (Films/Séries) tout en conservant la liste complète en mémoire.
- Bouton “Voir tout”: ajouté sur le header de chaque section, style cohérent avec Home (libellé, taille, espacement).
- Navigation “Voir tout”: vers une page dédiée `SearchResultsPage` qui reçoit `SearchResultsPageArgs { query, type }` où `type ∈ { movies, shows }`.
- SearchResultsPage: affiche l’intégralité des résultats pour le type sélectionné (pas de limite), réutilise les composants liste/grille existants.
- Providers: `searchResultsControllerProvider` (state: { query, type, items, isLoading, error }) qui consomme les use cases `SearchMovies`/`SearchShows`.
- Routage: ajouter `AppRouteNames.searchResults` et route `/search_results` dans `app_router.dart`.
- Non-usage de CategoryBrowser: ne pas pointer vers une catégorie (tout films/tout séries) car “Voir tout” doit conserver la pertinence de la requête et respecter la séparation des responsabilités.
- Étapes:
  1) Mettre à jour `SearchController` pour exposer `moviesTop10`/`showsTop10` et conserver `moviesAll`/`showsAll`.
  2) Ajouter le bouton “Voir tout” sur les headers Films/Séries de `SearchPage`.
  3) Créer `SearchResultsPage` + `SearchResultsPageArgs` + provider dédié.
  4) Ajouter la route et le nom de route dans le router.
  5) Vérifier navigation depuis Recherche → Voir tout (Films) et → Voir tout (Séries) avec la query courante.

3) DI & Infrastructure
- Mettre à jour `injector.dart` pour enregistrer `UserSettingsRepositoryImpl` et `SearchHistoryRepositoryImpl`.
- Paramétrage timeouts/retry réseau côté data (où applicable); pas de retry infini.
- Assurer null-safety et valeurs par défaut sûres dans tous les états UI.

4) Journalisation & erreurs
- Centraliser mapping exceptions → `Failure` (storage, network, parse).
- Sur UI: afficher messages sobres (“Échec du chargement”, “Pas de résultats”), sans exposer payload sensibles.

5) Checklist livraison
- Code formaté (`dart format`), lints OK (`flutter_lints`).
- Navigation: Bienvenue Étape1 → Étape2 (ajout source), Search route accessible.
- Test manuel des cas: pas de résultats, résultats nombreux, suppression historique, reprise app (profil chargé).
