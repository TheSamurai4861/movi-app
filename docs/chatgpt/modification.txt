# Diagnostic

* La requête **Trending** est bien exécutée par le repo, mais la **liste appliquée au Hero peut rester vide** à cause de filtres/mappings en chaîne dans `HomeFeedRepositoryImpl.getHeroMovies()` :

  1. on cherche d’abord des tendances **qui matchent l’IPTV** et **avec poster** (`dto.posterPath != null && availableTmdb.contains(dto.id)`), sinon on tombe en **fallback** sur la première page trending, **puis** on filtre encore à l’étape de mapping si **`_images.poster(dto.posterPath)` est null** → l’élément est écarté. Résultat : la méthode peut renvoyer `[]` **sans erreur**, et le Hero n’affiche rien (le contrôleur met juste `hero: []`). 
  2. Le contrôleur `HomeController.load()` applique tel quel la liste reçue ; s’il n’y a **aucune erreur**, il publie un `state` avec `hero` vide, donc le widget Hero reçoit `null` et affiche un **placeholder** (ou rien de visible). 

* Chaîne de propagation (OK mais “silencieusement vide”) :
  `TmdbMovieRemoteDataSource.fetchTrendingMovies(...) → HomeFeedRepositoryImpl.getHeroMovies() → HomeController.load() (state.hero=...) → HomePage → HomeHeroCarousel | HomeHeroSection`. La chaîne **n’interrompt pas** en cas de liste vide. 

* Filtrages/tri :

  * **Match IPTV + poster requis** en priorité (peut éliminer toutes les entrées si `availableTmdb` est vide ou si `posterPath` manquant). 
  * Fallback trending **re-filtré** au mapping si le resolver image renvoie `null`. Aucun tri par score/popularité explicite côté repo. 

* Pagination/`time_window` : fenêtre **`week`** et pagination **1→3**. Si les pages renvoient des éléments sans poster exploitable, on sort avec `[]`. 

* Conversions DTO→domain : `_MovieLiteDto.tryParse` exige `id` + `title`; `poster_path`/`backdrop_path` sont optionnels, mais **le mapping final écarte l’item si le poster résolu est `null`**. Pas de `media_type` (flux “movie” uniquement). 

* États ViewModel → Hero : `HomeController.load()` positionne `isLoading=true`, puis **publie `hero` même si vide** et remet `isLoading=false`. Pas de passage par un `empty` explicite ; donc **state vide = pas d’affichage**. 

* Cache : le **cache TMDB** n’est pas utilisé pour la construction de la liste “trending → hero”, seulement pour l’**enrichissement** des catégories IPTV (et là, la sélection d’image utilise `_PosterSelector`). Donc le cache **n’explique pas** l’absence de trends au Hero. 

* Régressions possibles vues dans le code :

  * **Sélection d’image** : pour le Hero, on utilise `TmdbImageResolver.poster/backdrop` directement (pas `_PosterSelector`), ce qui **durcit** la condition d’acceptation (un resolver “strict” peut renvoyer `null` et vider la liste). 
  * Plusieurs `catch(_) {}` silencieux : les erreurs réseau/mapping sont absorbées ; on finit en **state vide** sans signaler l’anomalie à l’UI. 

# Flux & responsabilités

`home_hero_section` (ou `home_hero_carousel`) lit `state.hero` depuis `HomeController` → `HomeController.load()` demande `repo.getHeroMovies()` → le repo :

1. tente **Trending ∩ IPTV avec poster** (pages 1..3),
2. sinon **fallback Trending page 1** filtré au mapping par **poster résolu non-null**,
3. sinon **fallback VOD** (premier item playlist).
   Le **choix de la source** et l’**écrasement** éventuel sont **dans le repo** ; le ViewModel ne modifie pas la sélection. Si toutes ces étapes produisent `[]`, le **Hero est ignoré** côté UI (pas de remplacement). 

# Analyse données / mapping / configuration

* Endpoint “trending” : appelé via `TmdbMovieRemoteDataSource.fetchTrendingMovies(window:'week', page:n)` → renvoie `List<dynamic>` consommée par `_MovieLiteDto.tryParse`. (On ne voit pas ici l’URL exacte, mais le **paramètre `window`** est bien transmis et **la pagination** gérée.) 
* DTO/mappers : `id`, `title|original_title`, `poster_path`, `backdrop_path`, `release_date`. L’étape critique est **l’exigence d’un poster résolu** au mapping final. 
* `image_url_resolver` : le repo appelle `TmdbImageResolver.poster/backdrop` (non fourni ici). Si son comportement retourne `null` pour certaines tailles/chemins, **les items sont écartés**. Pour l’enrichissement IPTV (pas le Hero), il existe un `_PosterSelector` plus robuste, **non utilisé** pour le Hero. 
* Cache : non impliqué pour le Hero trending → aucune clé/TTL à incriminer dans ce flux. 

# UI / comportement (si applicable)

* Lecture du state : `HomePage` → `homeControllerProvider`. Si `state.hero.length ≥ 2` → `HomeHeroCarousel`, sinon `HomeHeroSection(movie: state.hero.isNotEmpty ? state.hero.first : null)`. Quand `hero=[]`, le **widget reçoit `null`**. Pas de réapplication/fallback UI. 
* Re-renders : le contrôleur met `isLoading` puis publie `hero`; aucun `notifyListeners` manquant (Riverpod `StateNotifier`), la mise à jour est bien propagée. 
* Logique locale : pas de filtre/tri côté widget ; toute la sélection vient du repo. 

# Pistes de résolution (sans code)

* **Centraliser** la sélection d’images pour le Hero sur la **même stratégie** que l’enrichissement (`_PosterSelector` ou équivalent) afin **d’éviter d’écarter** des éléments valides pour cause de poster non résolu par le resolver. 

* Dans `getHeroMovies()`, **tolérer** un fallback sémantique (ex. backdrop si poster absent, ou poster IPTV si disponible) avant d’exclure un item. Aujourd’hui, tout élément sans `poster` résolu est **jeté**. 

* Ajouter un **state explicite `empty`** côté contrôleur (ou au moins un flag) et un **message/log** quand `hero=[]` pour éviter un “no-op” silencieux. 

* **Réduire la dépendance** au match IPTV pour la première passe : si `availableTmdb` est vide, ne pas attendre la boucle sur 3 pages avant de basculer au fallback. 

* **Tracer** les raisons d’élimination (poster null, resolver null, pas dans IPTV, etc.) pour diagnostiquer les cas réels. 

* **Tests** à couvrir :

  * Trending 200 avec items → **non vide** après mapping (y compris cas poster manquant mais backdrop présent) → **Hero loaded**. 
  * Trending vide → **fallback VOD** présent → **Hero loaded**. 
  * Resolver retourne `null` pour posters → vérifier que la stratégie alternative (backdrop/selector) **remplit quand même** la liste. 
  * `availableTmdb` vide → ne pas passer tout le flux en “match IPTV” inutilement.
